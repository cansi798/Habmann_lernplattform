Repository & Execution

Vom Prozessbild zum steuerbaren System – Single Source of Truth (eine verbindliche Wahrheits-Quelle) + Workflow-Engine (Camunda-Style).

Repository steuert Transparenz. Execution steuert Durchlaufzeit. Beides braucht Owner.

Das große Bild · worum geht es heute?

In einem Satz: Wir machen aus einem Prozess-Bild ein laufendes System.

Stell dir vor

Bei deinem Arbeitgeber kommt jeden Tag eine Bestellung herein. Ein Mensch prüft sie, ein anderer gibt sie frei, eine Software bucht sie. Heute lernst du, wie diese Schritte nicht im Kopf einzelner Mitarbeitenden hängen, sondern als verbindliches Modell existieren — und wie dieses Modell eine Software direkt steuern kann.

1 · Repository

Eine zentrale Sammlung aller Prozesse als saubere Modelle — mit Owner (wer ist verantwortlich?), Status (Entwurf oder freigegeben?) und Version. Die Wahrheits-Quelle.

2 · Business-Modell

Dasselbe Prozessbild, aber so dargestellt, dass Fachbereich und Management es verstehen: grobe Boxen, Lanes für Rollen, keine technischen Details.

3 · Execution-Modell

Dasselbe Prozessbild, aber technisch präzise: Welche Daten? Welche Fehlerfälle? Welche System-Schnittstelle? Eine Workflow-Engine kann es direkt ausführen.

Am Ende des Tages weißt du: Welche Prozesse eignen sich für ein „laufendes" Modell — und welche besser auf Papier bleiben?

Lernziele · Tag 4

Was du am Ende des Tages konkret kannst — drei Stufen, die aufeinander aufbauen.

L1 · Wissen („Ich kenne…")

  • Repository als Single Source of Truth — die eine verbindliche Sammlung aller Prozessmodelle
  • 3 Repository-Ebenen: Value Chain → Subprozess → Artefakt
  • Den Unterschied zwischen Business-BPMN (Sprache fürs Verstehen) und Execution-BPMN (Vertrag fürs Ausführen)
  • Vier BPMN-Bausteine: User Task, Service Task, XOR-Gateway, Timer

Selbsttest: „Erklär einer fremden Person in einer Minute, was ein Process-Repository ist."

L2 · Anwendung („Ich kann tun…")

  • Repository-Struktur mit Pflicht-Metadaten (Owner, Status, Version) entwerfen
  • Freigabe-Workflow (Draft → Review → Approved → Retired) skizzieren
  • Einen automatisierbaren Slice schneiden — mit klarem Start, Daten und Fehlerfällen
  • Minimum-Daten für einen Workflow definieren (Was muss die Engine wissen?)

Selbsttest: „Schneide für einen realen Prozess deiner Firma einen 4-Wochen-Pilot-Slice."

L3 · Management („Ich entscheide…")

  • Variantenstrategie: Attribut-Varianten oder eigene Modelle?
  • Wie viel Governance ist nötig, ab wann wird sie Overhead?
  • Exception-Policy: bewusst MVP statt Perfektion
  • Irreversible Entscheidungen erkennen + schriftlich begründen

Selbsttest: „Entscheide einen Zielkonflikt heute — und sag warum."

Kernsatz heute: „Repository ohne Owner = Ablage. Execution ohne SLA = Hoffnung."

Repository · drei Ebenen reichen für den Start

Variantenfrage: als Attribute (kleine Unterschiede) oder als separate Modelle (echte E2E-Abweichungen)?

Freigabe-Workflow für Modelle

Draft → Review → Approved → Retired – mit klaren Rollen.

Modeler

Erstellt & pflegt. Versionierung. Keine Freigabe.

Reviewer

Prüft Qualität & Konsistenz. 4-Augen-Prinzip.

Owner

Trägt Inhalt & Wirkung. Genehmigt Approved.

BPMN-Bausteine · die vier Symbole, die heute reichen

Wenn du diese vier kennst, kannst du jedes Workflow-Diagramm lesen — auch die auf den nächsten Folien.

Merkregel · die vier Symbole in Worten

Orange Box = ein Mensch arbeitet (User Task)

Grüne Box = Software arbeitet (Service Task)

Raute mit × = hier wird entschieden, welcher Weg

Kreis mit Uhr = eine Frist läuft (Timer-Event)

Warum diese vier reichen

80 % aller realen Prozesse lassen sich mit genau diesen vier Bausteinen beschreiben. Das fünfte Symbol, das du oft siehst — ein Kreis ohne Inhalt — ist einfach Start bzw. Ende des Prozesses (dünner Rand = Start, dicker Rand = Ende).

Lies das Diagramm auf der nächsten Folie mit dieser Brille: orange = Mensch, grün = Software, Raute = Entscheidung, Uhr = Frist.

Business-BPMN vs. Execution-BPMN

Gleiches Werkzeug, zwei Detailgrade – zwei Verantwortungsbereiche.

AspektBusiness-BPMNExecution-BPMN (Camunda)
ZielgruppeFachbereich, ManagementEntwicklung, Workflow-Engine
Detailtiefegrob, Lanes & Tasks reichenpräzise, jede Variable definiert
Tasks„Rechnung prüfen“User Task vs. Service Task
Events„Erinnerung“Timer / Message / Error / Escalation
Datenim KopfVariablen, Schemas, APIs
Fehler„Sonderfall“Boundary Events (Begrenzungs-Ereignisse), Compensation (Rücknahme)
VerbindlichkeitKommunikationsmodellausführbarer Vertrag

Derselbe Prozess „Rechnung freigeben" — in beiden Detailgraden

Übergabeprinzip: Erst automatisierbaren Slice isolieren — klarer Start/Ende, klare Daten, klare Fehlerfälle. Dann erst aus Business-BPMN das Execution-BPMN ableiten.

Execution-BPMN · Rechnungsfreigabe

Mini-Camunda-Logik: User Task · Service Task · Timer · XOR.

Lies das Diagramm so · Schritt für Schritt

  1. Rechnung kommt rein (Start-Kreis links): per Mail-Postfach oder Upload-Portal.
  2. Daten extrahieren (grün/Service Task): Software liest Lieferant, Betrag, Posten.
  3. Erfassen prüfen (orange/User Task): Buchhaltung schaut drauf, korrigiert ggf.
  4. Betrag > 5.000? (Raute/XOR): ja → Teamlead-Freigabe nötig · nein → direkt weiter.
  5. Freigabe Teamlead (orange/User Task mit Timer): Timer wartet max. 3 Tage, danach Eskalation.
  6. Pfade führen wieder zusammen (Raute mit +).
  7. ERP buchen (grün/Service Task): API-Aufruf zum SAP-System.
  8. Fertig (End-Kreis rechts mit dickem Rand).

Was du in diesem Diagramm sehen kannst

  • Mensch vs. Software: 2 orange Schritte = Mensch, 2 grüne Schritte = Software.
  • Sicherheitsnetz: Der Timer auf der Teamlead-Freigabe verhindert, dass Vorgänge ewig liegen bleiben.
  • Zwei Wege: Kleine Beträge (< 5.000) gehen ohne Extra-Freigabe durch — Effizienz für 80 % der Fälle.
  • Daten fließen mit: Lieferant, Betrag, Kostenstelle reisen als Variablen durch die Engine — keine händische Übergabe.
Merksatz: Service Tasks (grün) = automatisiert. User Tasks (orange) = Menschen. Timer/Boundary = Sicherheitsnetz.

Automations-Slice schneiden

Drei Fragen, bevor man automatisiert.

1 · Klarer Start & Ende

Was triggert den Slice? Wann ist er fertig? Beides ohne „kommt drauf an“.

2 · Klare Daten

Welche Felder muss der Workflow zwingend kennen? (z. B. Betrag, KSt (Kostenstelle), Freigeber).

3 · Klare Fehlerfälle

Was tun, wenn Daten fehlen, API hängt, Timeout passiert?

Ausnahme-Politik MVP: 1–2 Sonderfälle bewusst draußen lassen – organisatorisch auffangen, später integrieren.

Senior-Brille: Beide Säulen parallel aufbauen – Repository ohne Execution = Papier, Execution ohne Repository = Insel.

Fallstudie · MediSupply Services

8 Wochen, 90k Budget, mehrere Standorte mit Variantenwunsch, unklare ERP-Schnittstelle.

Pilot-Auswahl Begründung

  • Hoher Standardisierungs-Hebel
  • Audit-Relevanz nachweisbar
  • Eine ERP-Schnittstelle reicht für MVP
  • Subprozess automatisierbar in 8 Wochen

Automations-Slice (Beispiel)

  • Start: Bestellung im Portal eingegangen
  • User Task „Plausibilität“ · Service Task „Bonitätscheck“
  • Eskalation nach 48h Inaktivität
  • Bewusst ausgespart: Sonderkonditionen → manuell
KPIs für Steuerung: Durchlaufzeit · Eskalationsquote · Rework. Je KPI ein Owner mit Eskalationspfad.

Coaching · vier konkrete Situationen

Stell dir vor — was tust du in diesem Moment? Vier Mini-Szenarien, vier Antworten.

Szenario: Repository wird zur Ablage

Beobachtung: Im Repository liegen 200 BPMN-Diagramme. Aber niemand pflegt sie. Letzte Änderung: vor 2 Jahren.

Antwort: Owner pro Modell festlegen. Quartalsweise Review-Routine. Modelle ohne Owner nach 6 Monaten in „Retired" verschieben.

Szenario: Engine ohne Vertrag

Beobachtung: Workflow läuft seit 3 Monaten. Mitarbeitende klagen über Hänger. Keiner weiß, wer für Fehler zuständig ist.

Antwort: SLA definieren (Reaktionszeit, Lösungszeit). Eskalationspfad: 1st-Level → 2nd-Level → Process Owner.

Szenario: Übergabe zerschellt

Beobachtung: Fachbereich übergibt Modell an IT. IT sagt: „So können wir das nicht ausführen." Fachbereich: „Aber genau das wollen wir!"

Antwort: Übergabe-Termin vor dem Modellieren. Business-BPMN gemeinsam reviewen. Erst dann Execution-BPMN ableiten.

Szenario: MVP läuft zu kurz

Beobachtung: 2 Sonderfälle wurden bewusst ausgespart. Nach 4 Wochen Pilot: niemand bemerkt sie — alle drücken auf Skalierung.

Antwort: Frühwarnsignale ansehen (Eskalationsquote, Rework, Audit-Findings). Erst skalieren, wenn alle drei Signale grün sind.

Management · vier Zielkonflikte

Pro Konflikt: die Spannung — und eine konkrete Leitfrage, die hilft zu entscheiden.

Standard ↔ Standortautonomie

Guardrails statt Wildwuchs — aber wo darf lokal entschieden werden?

Leitfrage: „Standort München will eine eigene Sonderfreigabe einbauen. Lassen wir das zu — oder zwingen wir den Standardprozess durch?"

Repository-Tiefe ↔ Pflegeaufwand

Vollständig ist nicht hilfreich — hilfreich ist nicht vollständig.

Leitfrage: „Modellieren wir nur den Happy Path oder alle 12 Ausnahmen? Wer pflegt die Ausnahmen in 2 Jahren noch?"

Automation ↔ Audit-Sicherheit

Schnelle Automation kann Compliance untergraben — Boundary Events sind nicht optional.

Leitfrage: „Wenn der Bot eine falsche Buchung macht — wie viele Klicks weit ist der Audit Trail davon entfernt?"

Speed ↔ Reversibilität

Welche Architektur-Entscheidung kostet in 6 Monaten am meisten beim Rollback?

Leitfrage: „Wenn wir nach 6 Monaten zurück müssen — gibt es einen Plan B, oder müssen wir alles neu bauen?"

Faustregel: Irreversible Entscheidungen brauchen schriftliche Begründung + Frühwarnsignal — sonst wird sie still und teuer.

Der rote Faden · vom Bild zum laufenden System

Fünf Stufen — jede einzelne hast du heute kennengelernt. Hier nochmal in der richtigen Reihenfolge.

Repository → Business-Modell

Im Repository liegt eine Prozesslandkarte mit Owner und Freigabe-Status. Aus einer Box dieser Landkarte wird ein Business-BPMN: Fachbereich liest, Management entscheidet.

Business → Execution-Modell

Aus dem Business-Modell wird ein Execution-BPMN: Welcher Schritt ist Mensch (User Task), welcher Software (Service Task)? Welche Daten fließen? Welche Fehlerfälle gibt es?

Execution → Engine → Steuerung

Eine Workflow-Engine (z.B. Camunda) führt das Execution-Modell direkt aus. KPIs wie Durchlaufzeit und Eskalationsquote zeigen, ob es funktioniert — Erkenntnisse fließen zurück ins Modell.

Kein Schritt ist optional. Skip-Risiken: Repository fehlt = Wissen geht verloren · Business-BPMN fehlt = Fachbereich versteht Execution nicht · Execution-BPMN fehlt = Engine läuft ins Leere.

Take-aways · fünf Merksätze

✓Repository ohne Owner = Ablage. Execution ohne SLA = Hoffnung.
✓Drei Ebenen reichen für den Start: Value Chain (Wertschöpfungskette) → Prozess → Artefakt.
✓Business-BPMN ist Sprache. Execution-BPMN ist Vertrag.
✓Schnitt vor Implementierung: Start, Daten, Fehler klären.
✓MVP statt Perfektion – mit Frühwarnsignal & Rückbau-Plan.

Fachbegriffe & Definitionen

Alle Abkürzungen und Fachbegriffe dieser Präsentation auf einen Blick.

BPMN

Business Process Model and Notation

Def.: Internationaler Standard (ISO 19510) für die grafische Darstellung von Geschäftsprozessen.

Einfach: Die genormte Bildsprache für Prozessdiagramme — damit alle dasselbe Bild lesen.

MVP

Minimum Viable Product

Def.: Erste lauffähige Produktversion mit gerade so vielen Funktionen, dass echtes Nutzer-Feedback möglich wird.

Einfach: Die kleinste Version, die echtes Feedback liefert — kein Schaufenster-Prototyp.

SLA

Service Level Agreement

Def.: Vertraglich vereinbarte Dienstleistungsqualität zwischen Anbieter und Kunde (Verfügbarkeit, Reaktionszeit, Lösungszeit).

Einfach: Schwarz auf weiß: Was muss der Dienst leisten, wann wird es kritisch, wer haftet?

KPI

Key Performance Indicator (Plural: KPIs)

Def.: Quantitativ messbare Kennzahl zur Bewertung des Zielerreichungsgrads einer Aktivität, eines Prozesses oder einer Organisation.

Einfach: Die wenigen Zahlen, an denen man WIRKLICH ablesen kann, ob ein Ziel erreicht ist.

XOR

Exclusive Or (Exklusives Oder)

Def.: Logischer Operator: wahr, wenn genau eine von mehreren Bedingungen wahr ist.

Einfach: Entweder A oder B — aber nicht beide gleichzeitig. Im BPMN: die Entscheidungs-Raute.

ERP

Enterprise Resource Planning

Def.: Integriertes Softwaresystem zur Planung und Steuerung unternehmensweiter Ressourcen (z.B. SAP, Oracle, Dynamics).

Einfach: Das Rückgrat-IT-System, in dem Buchhaltung, Einkauf, Lager und HR im selben Datenbestand laufen.

API

Application Programming Interface

Def.: Standardisierte Programmierschnittstelle, über die Softwaresysteme strukturiert Daten austauschen.

Einfach: Die „Steckdose", an die andere Programme andocken, um etwas anzufragen oder zu liefern.

BPMS

Business Process Management System

Def.: Softwareplattform zur Modellierung, Ausführung, Überwachung und Optimierung von Geschäftsprozessen.

Einfach: Die Werkzeugkiste, die ein BPMN-Diagramm zu einem laufenden Prozess macht.

IT

Informationstechnologie

Def.: Gesamtheit von Hardware, Software, Netzwerken und Diensten zur Verarbeitung und Übertragung von Information.

Einfach: Alles, was an Rechnern, Servern, Software und Netzen zum Job dazugehört.

EA

Enterprise Architecture

Def.: Gesamtsicht auf die Strukturen eines Unternehmens (Geschäft, Anwendungen, Technologie) als Steuerungsrahmen.

Einfach: Die Landkarte, die zeigt, wie Strategie, Prozesse, IT und Infrastruktur zusammenspielen.

MDA

Model-Driven Architecture

Def.: OMG-Standard zur Softwareentwicklung über plattformunabhängige Modelle, die durch Transformation in lauffähigen Code überführt werden.

Einfach: Erst Modell, dann automatisch der Code — Modelle sind die Wahrheit, nicht die Quelldatei.
