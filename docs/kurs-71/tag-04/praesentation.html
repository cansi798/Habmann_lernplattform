Repository & Execution

Vom Prozessbild zum steuerbaren System – Single Source of Truth + Workflow-Engine (Camunda-Style).

Repository steuert Transparenz. Execution steuert Durchlaufzeit. Beides braucht Owner.

Repository · drei Ebenen reichen für den Start

Variantenfrage: als Attribute (kleine Unterschiede) oder als separate Modelle (echte E2E-Abweichungen)?

Freigabe-Workflow für Modelle

Draft → Review → Approved → Retired – mit klaren Rollen.

Modeler

Erstellt & pflegt. Versionierung. Keine Freigabe.

Reviewer

Prüft Qualität & Konsistenz. 4-Augen-Prinzip.

Owner

Trägt Inhalt & Wirkung. Genehmigt Approved.

Business-BPMN vs. Execution-BPMN

Gleiches Werkzeug, zwei Detailgrade – zwei Verantwortungsbereiche.

AspektBusiness-BPMNExecution-BPMN (Camunda)
ZielgruppeFachbereich, ManagementEntwicklung, Workflow-Engine
Detailtiefegrob, Lanes & Tasks reichenpräzise, jede Variable definiert
Tasks„Rechnung prüfen“User Task vs. Service Task
Events„Erinnerung“Timer / Message / Error / Escalation
Datenim KopfVariablen, Schemas, APIs
Fehler„Sonderfall“Boundary Events, Compensation
VerbindlichkeitKommunikationsmodellausführbarer Vertrag
Übergabeprinzip: erst automatisierbaren Slice isolieren – klarer Start/Ende, klare Daten, klare Fehlerfälle.

Execution-BPMN · Rechnungsfreigabe

Mini-Camunda-Logik: User Task · Service Task · Timer · XOR.

Service Tasks (grün) = automatisiert. User Tasks (orange) = Menschen. Timer/Boundary = Sicherheitsnetz.

Automations-Slice schneiden

Drei Fragen, bevor man automatisiert.

1 · Klarer Start & Ende

Was triggert den Slice? Wann ist er fertig? Beides ohne „kommt drauf an“.

2 · Klare Daten

Welche Felder muss der Workflow zwingend kennen? (z. B. Betrag, KSt, Freigeber).

3 · Klare Fehlerfälle

Was tun, wenn Daten fehlen, API hängt, Timeout passiert?

Ausnahme-Politik MVP: 1–2 Sonderfälle bewusst draußen lassen – organisatorisch auffangen, später integrieren.

Repository vs. Execution · Steuerungs-Unterschied

DimensionRepositoryExecution (BPMS)
HebelTransparenz, Konsistenz, AuditDurchlaufzeit, Qualität, Audit Trail
Wirkunglangsam, organisationsweitschnell, prozessspezifisch
Risikotote Ablage ohne OwnerEngine ohne Modell = teurer Schrott
OwnerProcess OwnerAutomation Owner + IT
InvestitionsschutzStandards, Naming, LifecycleVersionierung, Migration, Rollback
Senior-Brille: Beide Säulen parallel aufbauen – Repository ohne Execution = Papier, Execution ohne Repository = Insel.

Fallstudie · MediSupply Services

8 Wochen, 90k Budget, mehrere Standorte mit Variantenwunsch, unklare ERP-Schnittstelle.

Pilot-Auswahl Begründung

  • Hoher Standardisierungs-Hebel
  • Audit-Relevanz nachweisbar
  • Eine ERP-Schnittstelle reicht für MVP
  • Subprozess automatisierbar in 8 Wochen

Automations-Slice (Beispiel)

  • Start: Bestellung im Portal eingegangen
  • User Task „Plausibilität“ · Service Task „Bonitätscheck“
  • Eskalation nach 48h Inaktivität
  • Bewusst ausgespart: Sonderkonditionen → manuell
KPIs für Steuerung: Durchlaufzeit · Eskalationsquote · Rework. Je KPI ein Owner mit Eskalationspfad.

Coaching · Verantwortungsfähigkeit

Repository ist Betriebssystem

Ohne Owner/Review/Lifecycle wird es zur Ablage – nicht zur Steuerung.

Execution ist Vertrag

Daten, Fehler, SLAs, Eskalation – wer trägt das Risiko falscher Automatisierung?

Saubere Übergabe

Business-Sicht (verständlich) ↔ Execution-Sicht (präzise). Zwei Modelle, ein Auftrag.

MVP statt Perfektion

Aber mit Frühwarnsignalen: Eskalationsquote, Rework, Audit-Findings.

Management · vier Zielkonflikte

Standard ↔ Standortautonomie

Guardrails statt Wildwuchs – aber wo darf lokal entschieden werden?

Repository-Tiefe ↔ Pflegeaufwand

Vollständig ist nicht hilfreich – hilfreich ist nicht vollständig.

Automation ↔ Audit-Sicherheit

Schnelle Automation kann Compliance untergraben – Boundary Events sind nicht optional.

Speed ↔ Reversibilität

Welche Architektur-Entscheidung kostet uns in 6 Monaten am meisten beim Rollback?

Irreversible Entscheidungen brauchen schriftliche Begründung + Frühwarnsignal.

Take-aways · fünf Merksätze

✓Repository ohne Owner = Ablage. Execution ohne SLA = Hoffnung.
✓Drei Ebenen reichen für den Start: Value Chain → Prozess → Artefakt.
✓Business-BPMN ist Sprache. Execution-BPMN ist Vertrag.
✓Schnitt vor Implementierung: Start, Daten, Fehler klären.
✓MVP statt Perfektion – mit Frühwarnsignal & Rückbau-Plan.
